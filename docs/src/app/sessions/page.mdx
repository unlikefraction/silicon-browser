# Sessions

Run multiple isolated browser instances:

```bash
# Different sessions
silicon-browser --session agent1 open site-a.com
silicon-browser --session agent2 open site-b.com

# Or via environment variable
SILICON_BROWSER_SESSION=agent1 silicon-browser click "#btn"

# List active sessions
silicon-browser session list
# Output:
# Active sessions:
# -> default
#    agent1

# Show current session
silicon-browser session
```

## Session isolation

Each session has its own:

- Browser instance
- Cookies and storage
- Navigation history
- Authentication state

## Chrome profile reuse

The simplest way to reuse your existing login state: pass a Chrome profile name to `--profile`. silicon-browser copies the profile to a temp directory (read-only snapshot) and launches Chrome with your existing cookies and sessions.

```bash
# List available Chrome profiles
silicon-browser profiles

# Reuse your default Chrome profile's login state
silicon-browser --profile Default open https://gmail.com

# Use a named profile (by display name or directory name)
silicon-browser --profile "Work" open https://app.example.com

# Or via environment variable
SILICON_BROWSER_PROFILE=Default silicon-browser open https://gmail.com
```

## Persistent profiles

For a custom profile directory that persists state across browser restarts, pass a path to `--profile`:

```bash
# Use a persistent profile directory
silicon-browser --profile ~/.myapp-profile open myapp.com

# Login once, then reuse the authenticated session
silicon-browser --profile ~/.myapp-profile open myapp.com/dashboard

# Or via environment variable
SILICON_BROWSER_PROFILE=~/.myapp-profile silicon-browser open myapp.com
```

The profile directory stores:

- Cookies and localStorage
- IndexedDB data
- Service workers
- Browser cache
- Login sessions

## Import auth from your browser

If you are already logged in to a site in Chrome, you can grab that auth state and reuse it in silicon-browser. This is the fastest way to bypass login flows, OAuth, SSO, or 2FA.

**Step 1:** Start Chrome with remote debugging:

```bash
# macOS
"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --remote-debugging-port=9222

# Linux
google-chrome --remote-debugging-port=9222
```

Log in to your target site(s) in this Chrome window.

`--remote-debugging-port` exposes full browser control on localhost. Any local process can connect. Only use on trusted machines and close Chrome when done.

**Step 2:** Connect and save the authenticated state:

```bash
silicon-browser --auto-connect state save ./my-auth.json
```

**Step 3:** Use the saved auth in future sessions:

```bash
# Load auth at launch
silicon-browser --state ./my-auth.json open https://app.example.com/dashboard

# Or load into an existing session
silicon-browser state load ./my-auth.json
silicon-browser open https://app.example.com/dashboard
```

Combine with `--session-name` so the imported auth auto-persists across restarts:

```bash
silicon-browser --session-name myapp state load ./my-auth.json
# From now on, state auto-saves/restores for "myapp"
```

State files contain session tokens in plaintext. Add them to `.gitignore` and delete when no longer needed. For encryption at rest, see [State encryption](#state-encryption) below.

## Session persistence

Use `--session-name` to automatically save and restore cookies and localStorage across browser restarts:

```bash
# Auto-save/load state for "twitter" session
silicon-browser --session-name twitter open twitter.com

# Login once, then state persists automatically
silicon-browser --session-name twitter click "#login"

# Or via environment variable
export SILICON_BROWSER_SESSION_NAME=twitter
silicon-browser open twitter.com
```

State files are stored in `~/.silicon-browser/sessions/` and automatically loaded on daemon start.

### Session name rules

Session names must contain only alphanumeric characters, hyphens, and underscores:

```bash
# Valid session names
silicon-browser --session-name my-project open example.com
silicon-browser --session-name test_session_v2 open example.com

# Invalid (will be rejected)
silicon-browser --session-name "../bad" open example.com    # path traversal
silicon-browser --session-name "my session" open example.com # spaces
silicon-browser --session-name "foo/bar" open example.com    # slashes
```

## State encryption

Encrypt saved state files (cookies, localStorage) using AES-256-GCM:

```bash
# Generate a 256-bit key (64 hex characters)
openssl rand -hex 32

# Set the encryption key
export SILICON_BROWSER_ENCRYPTION_KEY=<your-64-char-hex-key>

# State files are now encrypted automatically
silicon-browser --session-name secure-session open example.com

# List states shows encryption status
silicon-browser state list
```

## State auto-expiration

Automatically delete old state files to prevent accumulation:

```bash
# Set expiration (default: 30 days)
export SILICON_BROWSER_STATE_EXPIRE_DAYS=7

# Manually clean old states
silicon-browser state clean --older-than 7
```

## State management commands

```bash
# List all saved states
silicon-browser state list

# Show state summary (cookies, origins, domains)
silicon-browser state show my-session-default.json

# Rename a state file
silicon-browser state rename old-name new-name

# Clear states for a specific session name
silicon-browser state clear my-session

# Clear all saved states
silicon-browser state clear --all

# Manual save/load (for custom paths)
silicon-browser state save ./backup.json
silicon-browser state load ./backup.json
```

## Authenticated sessions

Use `--headers` to set HTTP headers for a specific origin:

```bash
# Headers scoped to api.example.com only
silicon-browser open api.example.com --headers '{"Authorization": "Bearer <token>"}'

# Requests to api.example.com include the auth header
silicon-browser snapshot -i --json
silicon-browser click @e2

# Navigate to another domain - headers NOT sent
silicon-browser open other-site.com
```

Useful for:

- **Skipping login flows** - Authenticate via headers
- **Switching users** - Different auth tokens per session
- **API testing** - Access protected endpoints
- **Security** - Headers scoped to origin, not leaked

## Multiple origins

```bash
silicon-browser open api.example.com --headers '{"Authorization": "Bearer token1"}'
silicon-browser open api.acme.com --headers '{"Authorization": "Bearer token2"}'
```

## Global headers

For headers on all domains:

```bash
silicon-browser set headers '{"X-Custom-Header": "value"}'
```
